# 🧹 miracle_solework — Sepokat Data Pipeline

A personal data engineering portfolio project that ingests, transforms, and analyses order & expense data from **Sepokat** (a shoe-cleaning laundry business). The pipeline combines real transaction records with synthetically generated data, loads everything into **DuckDB**, transforms it with **dbt**, and exports results to **Google Sheets**.

## 🚀 Getting Started

### 1. Clone the repository

```bash
git clone https://github.com/<your-username>/miracle_solework.git
cd miracle_solework
```

### 2. Create & activate a virtual environment

```bash
python -m venv venv

# macOS / Linux
source venv/bin/activate

# Windows
venv\Scripts\activate
```

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

### 4. Set up environment variables

Copy the example below into a new `.env` file at the project root:

```env
# .env  (never commit this file)
GOOGLE_SHEETS_CREDENTIALS_JSON=path/to/your/service_account.json
SPREADSHEET_ID=your_google_sheet_id_here
```

> **Note:** `.env` is gitignored. Ask the project owner for the credentials if you need Google Sheets export.

### 5. Add raw data files

The `data/raw/` directory is gitignored (contains real business data). Place the following CSV files there before running the pipeline:

```
data/raw/
├── sepokat_2023.csv              # Real orders 2023
├── sepokat_2025_2026.csv         # Real orders 2025-2026
├── sepokat_spending_2023.csv     # Real expenses 2023
└── sepokat_spending_2025_2026.csv  # Real expenses 2025-2026
```

> **Note:** Ask the project owner for access to the raw data files.

## 🏃 Running the Pipeline

### Step 1 — Generate synthetic data

```bash
python pipeline/generate_synthetic.py
```

Outputs to `data/synthetic/`:
- `synthetic_orders.csv` — 300 synthetic order rows (Jan 2025 – Apr 2026)
- `synthetic_supplies.csv` — Recurring expense rows

### Step 2 — Ingest all data into DuckDB

```bash
python pipeline/ingest_to_duckdb.py
```

This script:
1. Reads all real + synthetic CSVs
2. Normalises & validates the data (date parsing, schema checks)
3. Loads `raw_orders` and `raw_expenses` tables into `data/warehouse/sepokat.duckdb`

### Step 3 — Run dbt transformations

```bash
cd miracle_solework_dbt
dbt run
dbt test
```

### Step 4 — Export to Google Sheets (optional)

```bash
python pipeline/export_to_sheets.py
```

## 🤝 How to Collaborate

### Branching Strategy

| Branch | Purpose |
|---|---|
| `main` | Stable, production-ready code |
| `feat/<name>` | New features (e.g. `feat/add-revenue-model`) |
| `fix/<name>` | Bug fixes (e.g. `fix/date-parsing`) |
| `chore/<name>` | Maintenance (e.g. `chore/update-deps`) |

### Contribution Workflow

1. **Fork** this repository or create a new branch from `main`
2. **Make your changes** on your branch
3. **Test locally** — make sure `ingest_to_duckdb.py` runs without errors
4. **Open a Pull Request** with a clear description of what changed and why
5. **Request a review** from the project owner before merging

### What to Work On

Check the [Issues](../../issues) tab for open tasks. Common areas:

- 🧱 **dbt models** — building staging/mart models in `miracle_solework_dbt/models/`
- 📊 **Analysis notebooks** — EDA in `analysis/`
- 🔧 **Pipeline improvements** — better error handling, logging, incremental loads
- 📤 **Google Sheets export** — `pipeline/export_to_sheets.py`

### Code Style Guidelines

- Python: follow **PEP 8** (use a formatter like `black` or `ruff`)
- Functions should have type hints and a short docstring
- Log meaningful messages using the existing `log` logger in each script
- Keep secrets out of code — always use `.env`

## 🔑 Profiles Setup (dbt)

dbt reads connection config from `profiles.yml` (gitignored). Create it at the project root:

```yaml
# profiles.yml  (never commit this file)
miracle_solework_dbt:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: "../data/warehouse/sepokat.duckdb"
```
